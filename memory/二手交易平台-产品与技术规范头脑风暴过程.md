## 三、方案探索

### 助手：架构方案对比

已收集足够的 MVP 核心功能需求，提出两种架构方案：

### 方案 A：轻量 MVP（推荐）

**核心思路**：Spring Boot 单体应用 + JSON 文件存储 + 原生前端，所有功能在同一服务中。文件上传用本地目录，WebSocket 直接内嵌。

**优点**：
- 部署简单，`java -jar` 即可运行，无需配置外部服务
- 与已有企业工单系统技术风格一致（分层架构、仓储接口抽象、文件持久化）
- 开发速度快，适合验证业务闭环

**缺点**：
- 文件存储不适合高并发，后续需迁移数据库
- 单体应用扩展性有限

### 方案 B：模块化 MVP

**核心思路**：前后端分离，后端用 Spring Boot + MySQL，前端独立部署（Nginx），WebSocket 独立管理。

**优点**：
- 更接近生产环境架构
- 性能更好，扩展性强

**缺点**：
- MVP 阶段过度设计，部署配置复杂
- 引入 MySQL 增加了环境依赖

**推荐方案 A**，目标是在 MVP 阶段快速验证从商品发布、沟通到订单完成的完整交易闭环。

### 助手：设计说明 2，核心业务流程与订单状态

#### 商品流转

每个商品为单库存，状态固定为：

```text
在售 → 已售/已下架
```

卖家可主动下架商品，商品被购买后自动变为"已售"。

#### 订单流转（担保交易模拟模式）

```text
待付款 → 已付款 → 已发货 → 已收货 → 已完成
  ↓         ↓
已取消 ←  已取消
```

- **待付款 → 已付款**：买家点击付款（模拟），系统记录付款时间
- **已付款 → 已发货**：卖家确认发货，填写物流信息或自提说明
- **已发货 → 已收货**：买家确认收货
- **已收货 → 已完成**：系统自动完成，或双方确认后完成，开启互评
- **取消规则**：
  - 待付款状态：买家可随时取消
  - 已付款状态：买家可申请取消，卖家同意后变为已取消；卖家在未发货前也可主动取消
  - 已发货之后不可取消
- **退款规则**：已收货后买家可申请退款，需卖家同意；若产生纠纷由管理员介入

#### 订单操作人明细

| 状态变更 | 操作人 |
| --- | --- |
| 待付款 → 已付款 | 买家 |
| 已付款 → 已发货 | 卖家 |
| 已发货 → 已收货 | 买家 |
| 已收货 → 已完成 | 系统自动 |
| 待付款 → 已取消 | 买家 |
| 已付款 → 已取消 | 卖家同意 / 卖家主动取消 |

#### 操作记录

系统必须记录每笔订单的状态变更操作人、操作时间和详情。订单 Service 层实现状态机校验，不允许非法状态跳转。该记录用于还原交易过程，并为纠纷处理提供依据。

#### 评价流程

交易完成后，买卖双方可给对方 1-5 星评分，评分展示在用户主页，作为信誉参考。

### 助手：设计说明 3，账号与权限规则

系统角色唯一：**普通用户**（同一账号可买可卖） + **管理员**。

| 角色 | 主要能力 |
| --- | --- |
| 普通用户 | 注册、登录、发布/管理商品、浏览/搜索商品、下单购买、站内聊天、确认收货、评价 |
| 管理员 | 管理用户（启用/禁用）、管理商品分类、处理纠纷（退款/退货争议）、查看全平台订单 |

#### 账号规则

- 注册仅需**用户名 + 密码**，登录后获得完整买卖权限
- 密码必须以安全哈希存储（BCrypt），不得明文落盘
- 管理员账号在系统初始化时创建，不开放注册。系统启动时检测是否存在管理员账号，若无则通过 `/api/init` 接口创建初始管理员
- 被禁用的用户不可登录、不可发布商品、不可发起聊天，但历史交易和商品数据保留
- 后端接口必须基于 JWT Token 实施权限验证，前端隐藏按钮不能替代后端授权检查

#### 前端按角色展示

- 普通用户：首页（商品浏览/搜索）、商品发布、我的商品、我的订单、聊天、个人主页（含评价）
- 管理员：除用户功能外，额外可见后台管理入口（用户管理、分类管理、纠纷处理）

### 助手：设计说明 5，技术架构

#### 整体架构

```text
原生 HTML/CSS/JS 页面
        |
Spring Boot REST API / 鉴权 / WebSocket
        |
      业务服务层
        |
      仓储接口层
        |
    JSON 文件存储适配器（MVP）

```

#### 后端分层

| 层 | 职责 |
| --- | --- |
| Controller | 接收 HTTP 请求，参数校验，返回 JSON；WebSocket 消息处理 |
| Service | 业务逻辑：商品发布、订单流转、聊天消息、评价计算 |
| Repository（接口） | 定义数据访问协议，与存储介质解耦 |
| JSON 适配器 | 实现 Repository 接口，读写 JSON 文件，处理并发写入保护 |

#### 前端结构

- 纯原生 JS，按页面划分脚本
- 路由通过前端简单的 hash 或路径判断实现
- WebSocket 客户端封装，负责聊天实时通信
- 商品图片通过 FormData 上传，后端存储到本地目录

#### 数据实体设计

| 实体 | 关键字段 | 说明 |
| --- | --- | --- |
| User | id、username、password(BCrypt)、role(user/admin)、enabled、createdAt | 用户账号，含角色标记 |
| Category | id、name、enabled、createdAt | 商品分类，管理员维护 |
| Product | id、sellerId、title、price、description、categoryId、tags、condition、tradeType、location、images[]、status(active/sold/off)、createdAt | 商品信息，单库存模式 |
| Order | id、productId、buyerId、sellerId、status、price、logisticsInfo、cancelReason、refundReason、createdAt、updatedAt | 订单，记录完整交易信息 |
| OrderLog | id、orderId、operatorId、action、detail、createdAt | 订单操作日志，追踪状态变更 |
| Message | id、senderId、receiverId、productId、content、isRead、createdAt | 聊天消息，关联商品 |
| Rating | id、orderId、raterId、ratedUserId、score(1-5)、createdAt | 交易评价，关联订单 |

#### 存储设计

- **JSON 文件持久化**：所有业务数据存储为 JSON 文件，服务重启后数据保留
- **商品图片**：本地目录 `uploads/{productId}/`，上传和读取
- **并发安全**：文件锁 + 原子写入，防止并发写入造成数据损坏
- **安全隔离**：`data/` 目录存放 JSON 数据文件，不得被 Spring Boot 静态资源挂载暴露

#### 仓储接口

所有 Service 层通过 Repository 接口访问数据，不得直接读写 JSON 文件：

- `UserRepository`
- `CategoryRepository`
- `ProductRepository`
- `OrderRepository`
- `MessageRepository`
- `RatingRepository`

每个接口均有 JSON 文件实现，预留数据库实现切换能力。Service 层通过构造注入依赖接口，Spring 通过配置决定注入哪种实现。

#### 关键技术决策

- **Spring Boot** + JDK 17+，嵌入式 Tomcat
- **BCrypt** 密码哈希
- **JWT Token** 身份认证（登录后签发，请求头携带）
- **WebSocket** 聊天，基于 Spring WebSocket（STOMP 协议）
- **Maven** 项目构建管理
- **文件锁/原子写入**保证 JSON 并发安全

## 四-B、设计变更与创新点确认

### 用户

感觉我们这个项目好像前端更适合用 Vue，原生代码会比较多，然后数据存储用 MySQL 不用 JSON。创新点我们加一个以物易物，同时你在合适的文档里添加功能需求和非功能需求。

### 助手：变更确认

收到，确认以下三项重大变更：

| 变更项 | 原方案 | 新方案 | 影响范围 |
| --- | --- | --- | --- |
| 前端技术栈 | 原生 HTML/CSS/JS | **Vue.js** | 全部前端页面改用 Vue 单文件组件，引入 Vue Router 路由 |
| 数据存储 | JSON 文件 | **MySQL** | 替换仓储实现，引入 Spring Data JPA，JSON 适配器方案废弃 |
| 创新功能 | 仅支持现金交易 | **新加以物易物** | 商品新增交易模式、以物易物流程、交换提议、双方确认 |

#### 以物易物功能要点

- 卖家发布商品时可选择交易模式：**出售** / **交换** / **两者均可**
- 标记为可交换的商品，买家可发起"以物易物"提议，从自己发布的在售商品中选择一件交换物
- 交换流程：
  1. 买家发起交换提议（选择自己的商品 → 填写交换意向说明）
  2. 卖家查看提议，可选择同意或拒绝
  3. 卖家同意后，双方各自发货
  4. 双方确认收货后交换完成
  5. 交换不涉及金额，或可设置补差价（MVP 阶段暂不实现差价）
- 交换订单与现金订单共用订单体系，通过订单类型区分

#### 技术栈变更影响

- **Vue.js**：前端改为 Vue 3 + Vue Router，页面以组件化方式组织，聊天使用 WebSocket 配合 Vue 响应式
- **MySQL**：实体类使用 JPA 注解，Repository 接口继承 JpaRepository，移除 JSON 适配器实现
- 仓储接口保留，但实现从 JSON 文件读写替换为 JPA 数据库操作
