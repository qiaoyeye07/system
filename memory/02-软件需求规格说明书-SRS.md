## 1. 引言

### 1.1 目的

本文档将已经确认的产品需求细化为可设计、可开发、可测试和可验收的软件需求。文档定义最小可行产品（MVP）的系统边界、用户角色、详细功能规则、输入校验、权限限制、订单状态约束、异常行为、数据留存和非功能要求。

本文档确认后，将作为页面与交互说明、技术设计说明书、数据模型设计、接口设计、测试与验收方案以及 MVP 实施计划的需求依据。

### 1.2 系统范围

系统为 C2C 个人对个人的二手交易平台，包含用户端和管理端。用户可注册登录、发布商品（支持出售和以物易物两种模式）、浏览搜索商品、实时聊天、下单购买或发起交换提议、管理订单并互评；管理员可管理用户、商品分类和裁定交易纠纷。

MVP 面向本地演示与小规模试用，使用浏览器访问，预计注册用户不超过 200 人、同时在售商品不超过 500 件。系统不在本阶段提供真实支付、物流追踪、第三方登录、消息推送、推荐算法或正式生产部署能力。

## 2. 总体描述

### 2.1 系统组成

系统应包含以下使用界面和服务能力：

| 组成部分 | 服务对象 | 主要能力 |
| --- | --- | --- |
| 用户端 | 普通用户 | 注册、登录、商品发布与管理、商品浏览与搜索、站内聊天、下单购买、订单管理、评价 |
| 管理端 | 管理员 | 用户管理、商品分类管理、纠纷处理、全平台订单查看 |
| 后端服务 | 全部界面 | 身份认证、授权、业务校验、数据访问、WebSocket 消息处理和记录保存 |
| 持久化组件 | 后端服务 | MVP 使用 JSON 文件保存数据，通过仓储接口抽象，支持未来更换存储实现 |

### 2.2 用户角色

| 角色 | 创建方式 | 数据访问范围 | 核心职责 |
| --- | --- | --- | --- |
| 普通用户 | 开放注册 | 本人的商品、订单、聊天；公开的商品和用户信息 | 发布商品、浏览购买、聊天沟通、管理订单、评价 |
| 管理员 | 系统初始化 | 全平台数据，包括所有用户、商品、订单和聊天记录 | 管理用户与分类、裁定纠纷 |

### 2.3 运行约束

- 前端使用 `Vue.js 3` 框架，通过 `Vite` 构建，配合 `Vue Router` 管理路由。
- 后端使用 Java `Spring Boot` 提供 REST 接口和 WebSocket 服务。
- 数据持久化使用 `MySQL` 数据库，通过 `Spring Data JPA` 操作。
- 站内聊天基于 WebSocket（STOMP 协议）实现实时通信。
- 身份认证基于 JWT Token。
- 密码使用 BCrypt 哈希存储。
- 商品图片存储于本地文件目录。
- 系统必须能够在本地环境启动并通过现代桌面浏览器使用。
- MVP 只要求支持小规模试用，不要求高并发或多实例部署。

### 2.4 范围外能力

以下能力不属于本 SRS 的 MVP 软件需求：

- 真实支付渠道接入（支付宝/微信支付）。
- 物流追踪 API 对接。
- 第三方登录（微信/支付宝等）。
- 消息推送/短信/邮件通知。
- 首页推荐流、瀑布流或智能推荐算法。
- 高级搜索与多条件筛选。
- 数据统计报表或管理看板。
- 商品页面公开留言/评论区。
- JSON 文件存储方式和仓储抽象层（已替换为 MySQL + JPA）。
- 以物易物中的补差价功能。
- 卖家编辑已发布商品的功能。

## 5. 认证与账号需求

### 5.1 用户注册

| 项目 | 说明 |
| --- | --- |
| 用例编号 | UC-AUTH-01 |
| 触发角色 | 未登录访客 |
| 前置条件 | 访客未登录；系统可正常保存数据。 |
| 输入信息 | 用户名、密码、确认密码。 |
| 成功结果 | 系统创建启用状态的用户账号；用户可使用该账号登录。 |
| 失败结果 | 系统不创建账号，并返回明确的校验或冲突提示。 |

#### 功能需求

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| SRS-AUTH-001 | 系统应向未登录访客提供用户注册能力。 | 必须 |
| SRS-AUTH-002 | 注册产生的账号角色必须固定为"普通用户"，访客不得在注册时选择管理员角色。 | 必须 |
| SRS-AUTH-003 | 注册成功后，新用户账号默认为启用状态。 | 必须 |
| SRS-AUTH-004 | 用户名在系统内必须唯一，比较时应忽略首尾空白。 | 必须 |
| SRS-AUTH-005 | 密码和确认密码不一致时，系统不得创建账号。 | 必须 |

#### 注册字段约束

| 字段 | 是否必填 | 约束 | 错误情形 |
| --- | --- | --- | --- |
| 用户名 | 是 | 去除首尾空白后长度为 3 至 50 个字符；允许中文、英文字母、数字、下划线和连字符。 | 为空、长度不合法、字符不合法或已存在。 |
| 密码 | 是 | 长度为 8 至 128 个字符。MVP 不强制复杂度组合。 | 为空或长度不合法。 |
| 确认密码 | 是 | 必须与密码完全一致。 | 与密码不一致。 |

### 5.2 登录与退出

| 项目 | 说明 |
| --- | --- |
| 用例编号 | UC-AUTH-02 |
| 触发角色 | 普通用户、管理员 |
| 前置条件 | 用户账号存在且为启用状态。 |
| 输入信息 | 用户名、密码。 |
| 成功结果 | 系统返回 JWT Token，并依据角色进入相应页面。 |
| 失败结果 | 系统不返回 Token，并提示登录失败。 |

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| SRS-AUTH-006 | 系统应允许用户使用用户名作为登录标识。 | 必须 |
| SRS-AUTH-007 | 密码验证失败、账号不存在或账号被禁用时，系统不得建立认证状态。 | 必须 |
| SRS-AUTH-008 | 对外登录失败提示不得暴露账号是否存在或是否已禁用，应统一提示"用户名或密码错误，或账号不可用"。 | 必须 |
| SRS-AUTH-009 | 已登录用户应能退出；退出后 Token 失效，不得继续访问受保护功能。 | 必须 |
| SRS-AUTH-010 | MVP 不提供找回密码、自助修改密码或登录失败锁定能力。 | 必须 |

### 5.3 初始化管理员与用户管理

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| SRS-AUTH-011 | 系统应提供初始化首个管理员账号的机制（通过 `/api/init` 接口）；该机制不得在已有管理员后继续允许创建。 | 必须 |
| SRS-AUTH-012 | 管理员账号不得开放注册。 | 必须 |
| SRS-AUTH-013 | 管理员应能查看用户列表，至少展示用户名、启用状态和注册时间。 | 必须 |
| SRS-AUTH-014 | 管理员应能将启用用户设置为禁用，并将禁用用户重新设置为启用。 | 必须 |
| SRS-AUTH-015 | 被禁用用户不得登录、发布商品、创建订单或发起聊天。 | 必须 |
| SRS-AUTH-016 | 用户被禁用后，其历史商品、订单、聊天记录和评价数据不得被删除。 | 必须 |

## 6. 商品分类需求

### 6.1 分类字段

| 字段 | 是否必填 | 约束 |
| --- | --- | --- |
| 分类标识 | 系统生成 | 全局唯一，创建后不得变化。 |
| 分类名称 | 是 | 去除首尾空白后长度为 1 至 50 个字符；在未删除的数据中不得重名。 |
| 状态 | 系统/管理员维护 | `启用` 或 `停用`；新建时默认为启用。 |
| 创建时间 | 系统生成 | 创建后不得修改。 |
| 更新时间 | 系统生成 | 编辑名称或状态时更新。 |

### 6.2 分类管理功能

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| SRS-CAT-001 | 仅管理员可以查看分类管理列表、创建分类、编辑分类名称或切换分类状态。 | 必须 |
| SRS-CAT-002 | 系统应预设若干默认分类（数码、服饰、家居、图书、其他）。 | 必须 |
| SRS-CAT-003 | 管理员可以停用分类；MVP 不提供物理删除分类功能。 | 必须 |
| SRS-CAT-004 | 卖家发布商品时，系统只应返回并接受处于启用状态的分类。 | 必须 |
| SRS-CAT-005 | 分类停用前已发布的商品仍应展示原分类名称，不因分类停用而失去关联信息。 | 必须 |
| SRS-CAT-006 | 普通用户可以在商品详情中查看分类，但不得维护分类。 | 必须 |

## 7. 商品需求

### 7.1 商品字段与约束

| 字段 | 产生方式 | 是否必填 | 约束 |
| --- | --- | --- | --- |
| 商品标识 | 系统生成 | 是 | 全局唯一，创建后不可更改。 |
| 标题 | 卖家输入 | 是 | 去除首尾空白后长度为 1 至 100 个字符。 |
| 价格 | 卖家输入 | 是 | 正数，单位为元，最多两位小数，最大 999999.99。 |
| 描述 | 卖家输入 | 是 | 去除首尾空白后长度为 1 至 4000 个字符。 |
| 分类 | 卖家选择 | 是 | 创建时必须引用一个有效分类。 |
| 标签 | 卖家输入 | 否 | 自定义标签，逗号分隔，最多 5 个，每个不超过 20 个字符。 |
| 成色 | 卖家选择 | 是 | 枚举：全新、几乎全新、有使用痕迹。 |
| 交易方式 | 卖家选择 | 是 | 枚举：自提、快递、两者均可。 |
| 所在地 | 卖家输入 | 是 | 去除首尾空白后长度为 1 至 50 个字符。 |
| 图片 | 卖家上传 | 否 | 最多 3 张，单张不超过 5MB，支持 jpg/png/gif 格式。 |
| 状态 | 系统维护 | 是 | 新建时为"在售"；购买后自动变为"已售"；卖家可手动变为"已下架"。 |
| 卖家 | 系统确定 | 是 | 固定为当前登录用户，不得由前端指定其他用户。 |
| 创建时间 | 系统生成 | 是 | 创建后不可修改。 |
| 更新时间 | 系统生成 | 是 | 状态变化时更新。 |

### 7.2 卖家发布商品

| --- | --- |
| 用例编号 | UC-PRD-01 |
| 触发角色 | 已登录且启用的用户 |
| 前置条件 | 至少存在一个有效商品分类。 |

| 成功结果 | 创建状态为"在售"的商品；该商品出现在首页和分类页。 |
| 异常结果 | 信息不符合约束、分类不存在或已停用、账号不可用时，不创建商品并返回原因。 |

| 编号 | 需求 | 优先级 |

| SRS-PRD-002 | 系统在保存商品前必须验证所选分类仍为有效状态。 | 必须 |
| SRS-PRD-003 | 创建成功时，系统应将商品状态设置为"在售"。 | 必须 |
| SRS-PRD-004 | MVP 不允许卖家编辑已发布商品的标题、价格、描述或分类，但允许下架操作。 | 必须 |
| SRS-PRD-005 | 卖家可以主动将在售商品下架，下架后商品不再公开展示。 | 必须 |
| SRS-PRD-006 | 商品被购买后，系统必须自动将状态变为"已售"。 | 必须 |

### 7.3 卖家商品管理

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| SRS-PRD-007 | 卖家应能查看自己发布的商品列表，按状态（在售/已售/已下架）筛选。 | 必须 |

## 11. 订单状态需求

### 11.1 状态转换矩阵

| 当前状态 | 允许的下一状态 | 允许操作者 | 前置条件 |
| --- | --- | --- | --- |
| 待付款 | 已付款 | 买家 | 无。 |
| 待付款 | 已取消 | 买家 | 无。 |
| 已付款 | 已发货 | 卖家 | 卖家填写物流/自提信息。 |
| 已付款 | 已取消 | 买家申请+卖家同意 | 买家申请取消；或卖家主动取消（未发货）。 |
| 已发货 | 已收货 | 买家 | 无。 |
| 已收货 | 已完成 | 系统自动/双方确认 | 收货后超时或手动确认。 |
| 已收货 | 退款中（纠纷） | 买家申请 | 买家填写退款原因。 |
| 已取消 | 无 | 无 | 终止状态。 |
| 已完成 | 无 | 无 | 终止状态；可进行评价。 |

### 11.2 状态处理规则

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| SRS-STS-001 | 订单状态集合必须限定为"待付款""已付款""已发货""已收货""已完成""已取消"。 | 必须 |
| SRS-STS-002 | 新创建订单的初始状态必须为"待付款"。 | 必须 |
| SRS-STS-003 | 状态只能依据状态转换矩阵向下一阶段推进，不允许跳级、回退。 | 必须 |
| SRS-STS-004 | 买家只能操作属于自己订单的状态推进（付款、收货、取消、退款）。 | 必须 |
| SRS-STS-005 | 卖家只能操作属于自己订单的发货和取消（已付款未发货时）。 | 必须 |
| SRS-STS-006 | 状态改变必须记录操作者、操作时间、原状态和新状态。 | 必须 |
| SRS-STS-007 | 订单进入"已完成"或"已取消"后禁止任何状态操作。 | 必须 |
| SRS-STS-008 | 取消后商品恢复"在售"状态，以便重新上架售卖。 | 必须 |

| --- | --- | --- |
| SRS-NFR-001 | 系统对失败操作应返回可理解的提示，以便用户修正输入或理解权限限制。 | 必须 |
| SRS-NFR-002 | 对涉及多项数据变更的单次操作，系统不得出现部分保存而页面显示成功的结果。 | 必须 |

## 16. 数据持久化与演进需求

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| SRS-DAT-001 | MVP 应将用户、分类、商品、订单、订单操作日志、聊天消息和评价数据持久保存于 MySQL 数据库中。 | 必须 |
| SRS-DAT-002 | 服务重新启动后，数据库中已保存的数据应能够继续读取和使用。 | 必须 |
| SRS-DAT-003 | 数据访问应通过 Spring Data JPA Repository 接口实现，业务服务通过 Service 层调用。 | 必须 |
| SRS-DAT-004 | 数据库连接和事务管理由 Spring 框架统一管理。 | 必须 |
| SRS-DAT-005 | 业务实体应使用自增 ID 或 UUID 作为主键，确保全局唯一。 | 必须 |
| SRS-DAT-006 | 系统首次启动时应自动执行 DDL 建表（spring.jpa.hibernate.ddl-auto=update），并初始化管理员账号。 | 必须 |
| SRS-DAT-007 | 商品图片持久化至本地文件目录 `uploads/{productId}/`，服务重启后可继续访问。数据库仅存储图片路径。 | 必须 |
| SRS-DAT-008 | 数据库连接信息（URL、用户名、密码）应配置在 `application.yml` 中，不得硬编码。 | 必须 |
| SRS-DAT-009 | 数据库表结构变更应通过 JPA 实体类定义体现，MVP 不要求使用 Flyway/Liquibase 等迁移工具。 | 必须 |

## 17. 安全需求

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| SRS-SEC-005 | 所有用户密码必须经过 BCrypt 安全哈希处理后再持久化，不得保存明文密码。 | 必须 |
| SRS-SEC-006 | 登录凭证（JWT Token）不得在页面中以明文密码形式保存或展示。 | 必须 |
| SRS-SEC-007 | 受保护数据的读取和写入均必须经过鉴权及权限检查。 | 必须 |
| SRS-SEC-008 | 用户输入的文本内容在页面展示时应防止被作为可执行脚本运行（XSS 防护）。 | 必须 |
| SRS-SEC-009 | MVP 无需实现密码找回、多因素认证、验证码或登录失败锁定。 | 必须 |
