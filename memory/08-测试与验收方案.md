| 项目 | 内容 |

| 编制日期 | 2026-07-08 |

## 1. 文档目的与范围

本文档定义二手交易平台 MVP 的测试策略、测试用例和验收标准，覆盖功能正确性、权限隔离、业务流程完整性、安全底线、数据持久化和可用性。本文档是 QA 测试、开发自测和 MVP 验收的直接依据。

**测试范围：**
- 功能测试：所有 PRD 功能需求的正向和异常路径
- 权限测试：角色隔离和资源归属校验
- 流程测试：现金交易和以物易物完整生命周期
- 安全测试：密码加密、JWT 鉴权、XSS 防护、越权访问
- 持久化测试：MySQL 数据重启后完整性
- 可用性测试：页面反馈、错误提示、空/加载/错误状态

**不在本方案范围：**
- 性能/压力测试
- 安全渗透测试
- 浏览器兼容性矩阵测试
- 移动端适配测试

## 2. 测试策略

### 2.1 测试分层

| 层级 | 范围 | 工具 | 执行阶段 |
| --- | --- | --- | --- |
| 单元测试 | Service 层业务逻辑、OrderStateMachine | JUnit 5 + Mockito | 开发中 |
| 接口测试 | 全部 REST API | Postman / REST Client | 接口开发完成后 |
| 集成测试 | 完整业务流程端到端 | 手工 + 浏览器 | 前后端联调后 |
| 验收测试 | 验收目标逐项确认 | 手工演示 | 提测完成后 |

### 2.2 测试环境

| 项目 | 要求 |
| --- | --- |
| JDK | 17+ |
| MySQL | 8.0+（测试库 `flea_market_test`） |
| 浏览器 | Chrome / Edge 最新版 |
| 测试数据 | 至少准备 3 个用户（含 1 个管理员）、5 个分类、10 件商品 |

| 编号 | 用例名称 | 前置条件 | 操作步骤 | 预期结果 |
| --- | --- | --- | --- | --- |
| TC-AUTH-01 | 正常注册 | 无 | 输入合法的用户名和密码，确认密码一致，点击注册 | 提示"注册成功，请登录"，跳转登录页。数据库新增用户，密码为 BCrypt 哈希。 |

| TC-AUTH-12 | 重复初始化管理员 | 已存在管理员 | 再次调用 POST /api/init | 返回 409，不创建第二个管理员。 |

## 5. 业务流程端到端测试

### 5.1 现金交易完整流程

```text
场景：用户 A 发布商品，用户 B 购买，完成交易并互评

1. 用户 A 注册 → 登录 → 发布商品（商品 ID=1，¥100，交易模式=仅出售）
   ✅ 商品出现在首页，状态为"在售"

2. 用户 B 注册 → 登录 → 浏览首页 → 搜索到商品 1 → 查看详情
   ✅ 详情显示完整信息和卖家评分

3. 用户 B 点击"联系卖家" → 发送"这个还在吗？"
   ✅ 用户 A 实时收到消息并回复

4. 用户 B 点击"立即购买"
   ✅ 订单创建（ORD-001），状态"待付款"，商品变为"已售"

5. 用户 B 进入订单详情 → 点击"付款"
   ✅ 状态变为"已付款"，操作记录更新

6. 用户 A 进入订单详情 → 点击"发货" → 填写 SF123456
   ✅ 状态变为"已发货"

7. 用户 B 点击"确认收货" → 二次确认
   ✅ 状态变为"已收货"

8. 系统等待 3 天（或手动调用完成）
   ✅ 状态自动变为"已完成"

9. 用户 B 评价用户 A：5 星 → 用户 A 评价用户 B：4 星
   ✅ 双方评价记录创建

10. 查看用户 A 主页 → 评分更新
    ✅ 显示 5.0 平均分，1 条评价
```

### 5.2 退款纠纷流程

```text
场景：用户 B 收货后发现商品有问题，申请退款被拒，管理员介入

1. 订单状态"已收货" → 用户 B 点击"申请退款" → 填写"商品有划痕"
   ✅ 退款申请提交

2. 用户 A 查看退款申请 → 点击"拒绝退款" → 填写"发货前拍照确认过"
   ✅ 状态变为"退款中"

3. 管理员登录 → 进入纠纷处理页 → 查看 ORD-001 纠纷详情
   ✅ 显示订单信息、双方陈述、聊天记录

4. 管理员选择"同意退款" → 填写裁定理由
   ✅ 订单变为"已取消"，商品恢复"在售"
```

### 5.3 以物易物完整流程

```text
场景：用户 A 发布可交换商品，用户 B 发起交换，双方完成交换

1. 用户 A 发布商品（交易模式=两者均可，商品 ID=10）
2. 用户 B 发布商品（交易模式=仅出售，商品 ID=20）

3. 用户 B 查看商品 10 → 点击"发起交换" → 选择商品 20 → 填写意向
   ✅ 交换订单创建（SWP-001），状态"待确认"

4. 用户 A 查看 SWP-001 → 点击"同意交换"
   ✅ 状态变为"已确认"，商品 10 和 20 自动下架

5. 用户 A 发货（SF-A） → 用户 B 发货（SF-B）
   ✅ 双方均发货后，状态变为"双方已发货"

6. 用户 A 收货 → 用户 B 收货
   ✅ 双方均收货后，状态变为"已完成"

7. 双方互评
   ✅ 评价创建成功
```

### 5.4 交换取消流程

```text
3. 用户 A 点击"同意取消"
   ✅ 状态变为"已取消"，双方商品恢复"在售"
```
