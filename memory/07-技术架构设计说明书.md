### 架构原则

| 原则 | 说明 |
| --- | --- |
| 分层隔离 | Controller → Service → Repository，上层只依赖下层接口。 |
| 状态机驱动 | 订单状态变更统一由 `OrderStateMachine` 校验，Service 不得绕过。 |
| 无状态服务 | 后端不存储会话状态，认证完全依赖 JWT Token。 |
| 前端路由守卫 | Vue Router 全局守卫拦截未登录和越权访问。 |
| 后端权限兜底 | 前端隐藏按钮仅优化体验，后端是唯一安全边界。 |

### 4.3 OrderStateMachine 设计

订单状态机是 Service 层的关键组件，所有订单状态变更必须经其校验：

```java
@Service
public class OrderStateMachine {

    // 现金订单状态转换映射
    private static final Map<OrderStatus, Set<OrderStatus>> CASH_TRANSITIONS = Map.of(
        PENDING_PAY,   Set.of(PAID, CANCELLED),
        PAID,          Set.of(SHIPPED, CANCELLED),
        SHIPPED,       Set.of(RECEIVED),
        RECEIVED,      Set.of(COMPLETED, DISPUTE),
        DISPUTE,       Set.of(CANCELLED, COMPLETED)
        // COMPLETED, CANCELLED 为终态，不在此映射中
    );

    // 交换订单状态转换映射
    private static final Map<OrderStatus, Set<OrderStatus>> SWAP_TRANSITIONS = Map.of(
        PENDING_CONFIRM, Set.of(CONFIRMED, REJECTED, CANCELLED),
        CONFIRMED,       Set.of(BOTH_SHIPPED, CANCELLED),
        BOTH_SHIPPED,    Set.of(COMPLETED)
        // COMPLETED, REJECTED, CANCELLED 为终态
    );

    public void validateTransition(Order order, OrderStatus targetStatus) {
        Map<OrderStatus, Set<OrderStatus>> transitions =
            order.getOrderType() == OrderType.CASH ? CASH_TRANSITIONS : SWAP_TRANSITIONS;

        Set<OrderStatus> allowed = transitions.get(order.getStatus());
        if (allowed == null || !allowed.contains(targetStatus)) {
            throw new BusinessException("当前状态不支持该操作");
        }
    }
}
```

## 5. 认证与安全方案

### 5.1 JWT 认证流程

```text
┌──────────┐                    ┌──────────────┐
│  客户端   │                    │  Spring Boot │
└────┬─────┘                    └──────┬───────┘
     │                                 │
     │  POST /api/auth/login           │
     │  {username, password}           │
     │ ──────────────────────────────→ │
     │                                 │ BCrypt 验证密码
     │                                 │ 生成 JWT (sub=userId, role, exp=24h)
     │  {token, user}                  │
     │ ←────────────────────────────── │
     │                                 │
     │  后续请求                        │
     │  Authorization: Bearer <token>  │
     │ ──────────────────────────────→ │
     │                                 │ JwtAuthenticationFilter
     │                                 │   ├─ 解析 Token
     │                                 │   ├─ 校验签名+过期
     │                                 │   ├─ 查询用户是否启用
     │                                 │   └─ 设置 SecurityContext
     │                                 │ Controller 处理请求
     │  {code:200, data:{...}}         │
     │ ←────────────────────────────── │
```

### 5.2 JWT Token 结构

```json
{
  "sub": "1",              // 用户 ID
  "username": "zhangsan",  // 用户名
  "role": "USER",          // 角色
  "iat": 1720425600,       // 签发时间
  "exp": 1720512000        // 过期时间（24h）
}
```

签名算法：HS256，密钥配置于 `application.yml`：

```yaml
app:
  jwt:
    secret: ${JWT_SECRET:your-256-bit-secret-key-here-minimum-32-chars}
    expiration-ms: 86400000  # 24 小时
```

### 6.3 路由导航守卫

```javascript
router.beforeEach((to, from, next) => {
  const token = localStorage.getItem('token');
  const user = store.user; // 从 Pinia/reactive 获取

  // 游客页面：已登录则跳首页
  if (to.meta.guest && token) {
    return next('/');
  }

  // 需认证页面：未登录跳登录页
  if (to.meta.auth && !token) {
    return next('/login');
  }

  // 管理员页面：角色校验
  if (to.meta.role === 'ADMIN' && user?.role !== 'ADMIN') {
    return next('/');
  }

  next();
});
```

### 6.4 Axios 拦截器

```javascript
// 请求拦截：自动携带 Token
api.interceptors.request.use(config => {
  const token = localStorage.getItem('token');
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});

// 响应拦截：统一错误处理
api.interceptors.response.use(
  response => response.data,
  error => {
    if (error.response?.status === 401) {
      localStorage.removeItem('token');
      store.clearUser();
      router.push('/login');
    }
    return Promise.reject(error.response?.data || error);
  }
);
```

## 7. WebSocket 通信架构

### 7.1 STOMP 消息流

```text
┌─────────────┐                         ┌──────────────────┐
│  Vue 客户端  │                         │   Spring Boot    │
│ (STOMP.js)  │                         │   WebSocket      │
└──────┬──────┘                         └────────┬─────────┘
       │                                         │
       │  CONNECT /ws/chat?token=<jwt>           │
       │ ──────────────────────────────────────→ │ JWT 校验
       │                                         │ 建立 Session
       │  CONNECTED                              │ 加入 OnlineUserTracker
       │ ←────────────────────────────────────── │
       │                                         │
       │  SUBSCRIBE /user/queue/messages         │
       │ ──────────────────────────────────────→ │ 订阅私聊通道
       │                                         │
       │  SEND /app/chat/send                    │
       │  {receiverId, productId, content}       │
       │ ──────────────────────────────────────→ │ ChatMessageHandler
       │                                         │   ├─ 校验用户 + 未禁用
       │                                         │   ├─ 保存到 message 表
       │                                         │   └─ 转发至接收方
       │                                         │
       │                                         │ /user/{receiverId}/queue/messages
       │                                         │ ← 推送给接收方
       │  MESSAGE {senderId, content, ...}       │
       │ ←────────────────────────────────────── │
```

## 9. 关键技术决策汇总

| 决策 | 选择 | 理由 |
| --- | --- | --- |
| 前后端分离 | Vue SPA + Spring Boot REST | 开发解耦，前端热更新，后端专注业务。 |
| 构建产物整合 | 前端构建到 `static/`，单 jar 部署 | MVP 简化部署，一个 `java -jar` 即可运行。 |
| 订单表统一 | 现金+交换共用一张表 | PRD 明确要求，减少表数量，简化关联查询。 |
| DDL 策略 | `ddl-auto=update` | MVP 阶段快速迭代，生产前改为 `validate` + Flyway。 |
| 图片存储 | 本地文件系统 | MVP 数据量小，无需对象存储。 |
| API 风格 | 统一 `ApiResponse` 包装 | 前端统一处理，错误信息可读。 |
| 前端状态管理 | 简易 reactive / Pinia | MVP 状态不复杂，避免过度设计。 |
| CSS 方案 | 组件内 `<style scoped>` | 组件隔离，无需额外 CSS 框架。 |
